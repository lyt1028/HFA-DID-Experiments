2. **实验与论文贡献不对齐**：论文核心贡献是(1)异步状态锚定范式(2)RA-ReDKG协议(3)多维信誉模型(4)三层联合验证，但实验未充分覆盖

5. **移动性攻击防御缺乏量化验证**：安全性分析流于理论，无仿真支撑
6. **自适应门限、乐观路径等核心机制未实验验证**

**服务器现状**：已有exp1-exp8完整实验框架，但以下关键机制尚未在代码中实现：

### 实验1：核心操作性能与可扩展性（对应贡献1+4）— 保留优化
- **目标**：量化DKG初始化、公钥聚合、凭证签发三阶段计算开销随委员会规模的变化
- **复用**：exp1a思路
- **自变量**：委员会节点数 n ∈ {4, 8, 12, 16, 20}
- **因变量**：各阶段平均计算时延(ms)
- **输出**：分组柱状图（图4）
- **要点**：展示系统在节点数增长时仍保持亚秒级性能

### 实验2：跨域验证性能对比（对应贡献1）— 保留并扩展域规模与跨域因素
- **目标**：对比HFA-DID与现有方案在签发和跨域验证的性能，**扩展到大规模跨域场景**，并考量多种跨域特有因素
- **复用**：exp1b + exp2c，域数量扩展到{2, 5, 10, 15, 20, 30}
- **子实验**：
  - 图5(a)：签发延迟对比（HFA-DID vs WeIdentity vs CanDID），自变量为节点数
  - 图5(b)：跨域验证延迟对比（HFA-DID vs Cross-Chain），自变量为域数量m∈{2, 5, 10, 15, 20, 30}
    - 重点关注m>10时HFA-DID的O(log m)增长 vs Cross-Chain的线性增长
    - 标注性能拐点
  - 图5(b)中延迟分解为：GCL快照查询 + Merkle验证 + BLS验签 + 网络传输，展示各阶段对总延迟的贡献比
- **跨域特有因素的实验覆盖**：
  1. **快照时效性（Snapshot Staleness）**：测试不同锚定频率（每1/3/5个epoch锚定一次）下，验证延迟与凭证状态新鲜度的权衡。锚定频率越低，GCL开销越小但快照越旧，验证者可能依赖过时的治理状态。在图5(b)的延迟分解中体现快照查询开销随锚定频率的变化。
  2. **域间异构性（Domain Heterogeneity）**：混合配置各域的委员会规模（小域n=4, 中域n=10, 大域n=20）和凭证批量大小（100/1000/5000），测试GCL聚合非均匀域根时的Merkle树构建开销与验证路径长度变化。
  3. **GCL聚合负载分离**：在30域场景下，将总延迟拆分为网络传输延迟和GCL端聚合计算延迟，明确大规模场景的真正瓶颈是网络I/O还是计算聚合。
  4. **并发验证吞吐**：测试多个验证者（10/50/100）并发查询GCL时的平均响应延迟和TPS上限，评估GCL作为验证中介的可扩展性。
- **输出**：图5(a)(b) + 表格或文字描述跨域因素分析结果
- **新增工作**：修改exp2a/exp2c参数支持最多30域；新增快照时效性与域间异构性的测试逻辑

### 实验3：RA-ReDKG轮转开销与稳定性（对应贡献2）— 保留并增加路径对比
- **目标**：评估RA-ReDKG的轮转开销，**对比乐观路径与完整路径的性能差异**
- **复用**：exp7c + exp7d
- **子实验**：
  - 图6(a)：轮转开销按阶段分解（零多项式生成、份额分发、份额聚合、一致性验证），自变量为n
  - 图6(b)：**乐观路径 vs 完整路径对比**
    - 自变量：留存率ρ ∈ {0.5, 0.6, 0.7, 0.8, 0.9, 1.0}
    - ρ≥ρ_th(0.7)时走乐观路径（仅留任节点参与，通信O(|Δ|·n)）
    - ρ<ρ_th时走完整路径（全体重新分发，通信O(n²)）
    - 因变量：轮转总耗时、通信量
- **输出**：两个子图并排（图6）
- **新增工作**：在redkg.py中实现乐观路径，对比完整路径

### 实验4：动态信誉机制有效性（对应贡献3）— 保留

- **复用**：exp4b
- **输出**：表1（签发成功率、恢复周期数对比）

### 🆕 实验6：自适应门限机制验证（对应贡献2+3，回应评审意见）
- **目标**：验证自适应门限t(e)在不同威胁态势下的动态调整行为及其安全-效率权衡
- **需新建实验脚本**
- **设计**：
  - 模拟20节点委员会，运行15个epoch
  - 在不同epoch注入不同程度的威胁（恶意节点比例0%→5%→15%→5%→0%）
  - 对比三种门限策略：
    1. **固定门限（高）**：t = ⌊n/2⌋ + 1 = 11（始终保守）
    2. **固定门限（低）**：t = ⌈n/3⌉ + 1 = 8（始终激进）
    3. **自适应门限**：t(e) = clamp(⌈n·(t_base + μν)⌉, t_min, t_max)，其中ν为信誉变异系数
- **因变量**：
  - 门限t(e)随epoch的变化曲线
  - 签发成功率（效率）
  - 安全裕度（t(e) - f_active）
- **输出**：图8（三条曲线对比门限变化 + 安全裕度）
- **实现**：
  - 新建 `src/models/adaptive_threshold.py`
  - 新建 `experiments/exp9_adaptive_threshold.py`

### 🆕 实验7：移动性攻击防御量化验证（回应评审"安全性分析流于理论"）
- **目标**：通过仿真验证定理5（移动性合谋攻击容忍界），量化三层防御机制的联合效果
- **需新建实验脚本**
- **设计**：
  - 模拟移动性攻击者以速率r持续腐蚀委员会节点
  - 自变量：攻击速率r ∈ {1, 2, 3, 4, 5}（每轮新腐蚀r个节点）
  - 委员会n=20, δ=0.8, θ=0.6 → k*=3, t_min=11 → 理论r_max=11/3≈3.67
- **对比三层防御的累加效果**：
  1. **仅门限隔离**（无信誉、无自适应）：固定t=11，被腐蚀节点不剔除，累积到≥t时沦陷
  2. **门限+信誉检测**（无自适应门限）：固定t=11，信誉机制k*=3轮后剔除恶意节点
  3. **完整三层**（门限+信誉+自适应）：t(e)动态升高
- **因变量**：
  - 各攻击速率下系统保持安全的epoch数
  - 不同防御层级下的安全边界
- **输出**：表2（不同r值 × 三种防御配置的安全持续时间/是否沦陷）
- **实现**：
  - 新建 `experiments/exp10_mobile_attack.py`（含MobileAdversarySimulator）
